## 相互递归
俩个不同函数相互调用的情况
例如一个例子:校验信用卡号的正确性
Luhn 算法:

这个算法的好处:如果有单个数字出错,结果就不会是10的倍数,,所以这个算法就是让结果是20的倍数的啊.

这个算法可以用一般的递归方式,同时要告诉俩个信息,一个是剩余的数,以及这是第几位
还有一种方式是用相互递归  相互递归的一个思想就是把状态给变成角色切换

### 相互递归的思想是什么？

一句话：__状态不存参数里，存进"你现在是哪个函数"里。__

展开说：

普通递归是一个人干活，靠参数记住"我在第几步"。 相互递归&#x662F;__&#x4E24;个人（或多个角色）交替干活__，"你是谁"本身就告诉你当前该做什么，不用记。

再往深一层：__它把"状态"变成了"角色切换"。__

- 普通递归：`pos` 参数 = 状态存在数据里
- 相互递归：odd/even 函数 = 状态存在控制流里

__什么时候用相互递归？__ 当问题天然有"交替"的节奏时——奇偶、黑白、你我、进出。硬要塞进一个函数用 if 判断也行，但相互递归让每种情况自成一体，结构更清晰。

更奇怪的递归？有，一个比一个怪
怪 1：树递归（tree recursion）——不只递归一次，递归好几次
之前所有递归，每层只调一次自己：


return n + sum_digits(n // 10)     # 一次递归调用
树递归，每层调多次自己：


def fib(n):
    if n <= 1:
        return n
    return fib(n-1) + fib(n-2)     # 两次递归调用!
为什么怪：之前递归是一条链（123→12→1），树递归是一棵树：


fib(4)
├── fib(3)
│   ├── fib(2)
│   │   ├── fib(1)
│   │   └── fib(0)
│   └── fib(1)
└── fib(2)
    ├── fib(1)
    └── fib(0)
一个问题分裂成两个子问题，像树枝分叉。这是后面 CS61A 的重点——计数问题、搜索问题全是树递归。

怪 2：尾递归（tail recursion）——递归调用是最后一个动作

def sum_digits(n, acc=0):
    if n < 10:
        return acc + n
    return sum_digits(n // 10, acc + n % 10)    # 递归是最后一个操作
为什么怪：之前"归"的阶段要做组合（n % 10 + sum_digits(...)），现在不用归了！因为结果用 acc 参数一路往下带，到底直接出答案，不需要回填。

对比：

普通递归	尾递归
递的阶段	拆分	拆分 + 累积结果
归的阶段	要组合	不用归，到底就是答案
两阶段？	有递有归	只有递，没有归
这打破了你"递归一定有两阶段"的认知——尾递归只有一个阶段。

怪 3：间接递归（indirect recursion）——A 调 B，B 调 C，C 调 A
相互递归是两个函数互相调（A↔B）。间接递归是一圈函数循环调：


A 调 B → B 调 C → C 调 A → A 调 B → ...
为什么怪：你单独看 A，它没有调自己。你单独看 B，也没有。但三个连起来形成一个环——递归藏在函数之间的关系里，不在单个函数内部。

现实类比：三个朋友传纸条，A 传给 B，B 传给 C，C 又传给 A。纸条转一圈回到 A 手里，这就是间接递归。

怪 4：不动点组合子（Y combinator）——没有名字也能递归
之前所有递归都靠函数有名字才能调用自己（def sum_digits 里调用 sum_digits）。

如果不给函数起名字呢？ 还能递归吗？

能。Y combinator 是一个"高阶函数"，你给它一个"还没递归的函数"，它帮你把它变成"能递归的函数"——不需要名字。


# 不用 def、不用名字,纯 lambda
Y = lambda f: (lambda x: f(x(x)))(lambda x: f(x(x)))
为什么怪：这是递归的"底层 hack"——它揭示了递归不依赖名字，依赖的是"函数能接收自己作为参数"。这个 CS61A 不会考，但它是最深层的"递归到底是什么"。

递归家族全景
类型	怪在哪	一句话
普通递归	不怪	自己调自己，有递有归
树递归	一次裂变多次	一个问题分裂成多个子问题
尾递归	没有归	结果往下带，到底就是答案
相互递归	状态在角色里	两个函数交替，不靠参数记状态
间接递归	环形调用	A→B→C→A，递归藏在关系里
Y combinator	不用名字	没名字也能递归，靠函数传自己
