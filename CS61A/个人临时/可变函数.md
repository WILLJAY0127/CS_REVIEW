1、CS61A 原版 withdraw 是干嘛的？
61A 用它专门讲一件事：
同一个函数，相同参数，多次调用，结果不一样 → 这就是可变函数（有状态、有副作用）
61A 标准最简原版代码
python
运行
def make_withdraw(balance):
    def withdraw(amount):
        nonlocal balance
        balance = balance - amount
        return balance
    return withdraw

python
运行

一模一样的参数 25，每次调用结果不同
✅ 这就是 典型可变函数 / 非纯函数

核心炸裂误区纠正：print 只是「看得见的副作用」，withdraw 才是「程序真正的副作用」
先给你 61A 官方定义（百分百贴合考点）：
副作用 = 函数调用后，改变了「函数外部的环境状态」，不是返回值本身
副作用 ≠ 打印输出
1. 为什么你误以为只有 print 是副作用？
print 是IO 副作用（可视化、最直观）
你能看见，所以你只认识它。
但 Python / CS61A 真正核心、考试最爱考的副作用是：
修改外部变量、修改状态
2. 重点：withdraw 根本没 print，为什么也有副作用？

分三层讲清楚：不是垃圾，是取舍；后端大量可变是客观刚需，但有规范能把失控压到最低
一、先回答核心：可变 / 副作用本身没有绝对好坏
1）纯函数（无副作用）优点
可缓存、可并行、好测试、输入固定输出固定，逻辑完全可控，不会出现 “上次调用影响这次结果”
适合计算、转换、校验工具方法
2）可变函数（带状态副作用）必须存在，后端根本绕不开
你的直觉没错：后端绝大多数业务都要修改持久状态（数据库、Redis、消息队列、文件、用户会话），而修改状态必然产生副作用：
用户注册 = insert、下单 = 新增订单、扣款 = 更新余额、退款 = 反向更新库存 / 资金
CS61A 的 withdraw 本质就是银行扣款，对应后端账户余额更新接口
只要业务需要留存变化，就必须用可变逻辑，纯函数做不了 “记录变更” 这件事。
3）为什么你会觉得 “代码容易垃圾、失控”？
问题不是可变本身，是无约束、无边界的滥用状态：
坏写法：全局变量满天飞，到处修改、多处读写同一个状态，调用顺序一变结果全乱；
规范写法：把状态收拢、隔离、加事务、分层，副作用集中管理，就不会失控。
二、为什么后端满眼都是可变逻辑，但大厂代码不会崩？4 套约束手段（对应 Java/Python 后端通用）
1. 状态隔离：把可变数据收拢，杜绝散落的全局变量
以 withdraw 余额举例：
❌ 垃圾写法：全局 balance，任意函数随便修改
python
运行
balance = 100
def withdraw(amt):
    global balance
    balance -= amt
随便哪个函数都能改余额，调用顺序乱、并发直接出错，完全不可控。
✅ 规范写法：状态封装，仅暴露受控操作（CS61A 闭包本质就是封装）
python
运行
def make_account(balance):
    def withdraw(amt):
        nonlocal balance
        if balance >= amt:
            balance -= amt
            return balance
        return "余额不足"
    def get_balance():
        return balance
    return withdraw, get_balance
# 外部只能通过 withdraw / get_balance 操作余额，无法直接篡改balance
w, query = make_account(100)
对应后端实践：数据库表、实体类只通过 Service 层 CRUD，Controller、工具类不准直接操作 SQL。
2. 分层：把「纯计算」和「可变 IO 操作」彻底拆分（核心工程规范）
所有后端项目标准分层：
工具层（全纯函数，无副作用）：金额计算、日期转换、参数校验、加密算法；
业务 Service 层（唯一允许存在副作用）：仅在这里写 DB/Redis 更新、扣款、下单；
禁止在工具、实体、Controller 里写任何修改持久状态的逻辑。
例子拆分扣款逻辑：
python
运行
# 纯函数（无副作用，只计算，不碰数据库）
def calc_new_balance(old_balance, deduct):
    return old_balance - deduct

# 可变函数（唯一允许改库，副作用集中一处）
def withdraw_service(account_id, deduct):
    old = db.query_balance(account_id)
    new_val = calc_new_balance(old, deduct)
    if new_val < 0:
        raise Exception("余额不足")
    db.update_balance(account_id, new_val)
    return new_val
计算逻辑永远纯净；修改数据库的副作用只集中在 Service，出问题只需要查这一层。
3. 并发与事务约束，解决多线程同时修改导致的混乱
可变最大风险：并发修改（两个人同时取钱，余额扣错）
后端解决方案：
数据库事务、行锁、乐观锁；
Redis 分布式锁；
把 “读取状态→修改→写入” 打包成原子操作，避免中间状态被篡改。
4. 幂等设计，削弱副作用带来的不可预测
纯函数天然幂等；可变操作要手动实现幂等，多次调用结果一致：
比如扣款接口增加唯一流水号，重复请求不会重复扣钱，大幅降低副作用带来的不确定性。
三、两种编程范式的取舍（帮你理解行业选择）
函数式编程（Haskell、Scala）：极力弱化可变，尽量只用纯函数，状态通过不可变数据传递；适合大数据计算、离线任务，但做线上业务系统会极度繁琐，每一次更新都要复制全量数据，性能差。
面向对象 / 命令式（Java/Python 后端主流）：允许可控的可变，牺牲一点点 “绝对纯净” 换取开发效率、存储性能；只要做好分层封装，完全能把复杂度锁死。
四、总结回答你的两个感受
后端全是可变是客观刚需：业务要存用户、订单、资金的变更，不可能全用纯函数；
可变不等于垃圾代码，随意散放、无隔离、无事务、不分层的可变代码才是垃圾；
最优实践：能写成纯函数的逻辑一律纯函数；必须修改状态的逻辑统一收拢到业务层，加锁、事务、封装隔离，限制副作用的扩散范围。
