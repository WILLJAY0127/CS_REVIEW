# 设计模式学习总结

---

## 🎯 核心认知：设计模式的本质

**你的理解（非常关键）**：更好地圈定稳定的范围，将稳定与不稳定因素做好限制与规划。

**精炼成业内标准表述**：

> 识别系统中「不变部分」和「变化部分」，通过抽象划定稳定边界，让变化被约束在指定区域，修改扩展不会污染稳定代码。

这本质上就是 **开闭原则（Open Closed Principle）**，也是所有 GoF23 模式共同的底层目标。

---

## ⚠️ 常见误区

| 误区 | 正确做法 |
|------|----------|
| 只背类图、角色、代码实现 | 建立「这个模式到底隔离了哪一块变化？」的思考习惯 |
| 为了可扩展而提前扩展 | 只有预判未来该位置存在变化时，才主动划分边界隔离 |
| 到处套用模式 | 如果确定逻辑永久不变，不需要额外抽象和模式 |

---

## 🏗️ 顶层认知框架

### 先学原则，再学模式

**原则是因，模式是果**。模式不是凭空创造的套路，是 SOLID + 迪米特法则 + 组合优于继承落地出来的成熟方案。

**学习顺序**：
1. 弄懂 5 大 SOLID（重中之重）
2. 理解「变化隔离」思想
3. 再逐个学习设计模式

### 学习模式的强制三连问

每次学习任意一个模式，强制自问：

| 问题 | 目的 |
|------|------|
| 它要隔离哪一类变化？ | 核心！找到模式的本质 |
| 它使用了什么抽象/边界来隔离？ | 理解实现手段 |
| 代价是什么？什么时候不该用？ | 避免过度设计 |

### 实例验证

| 模式 | 隔离的变化 | 隔离手段 | 代价 |
|------|-----------|----------|------|
| **策略模式** | 算法/业务规则经常变动 | 抽象策略接口，业务主体持有抽象 | 类数量变多；少量变动时属于过度设计 |
| **模板方法** | 固定流程中个别步骤实现不同 | 父类锁定稳定骨架，开放钩子方法留给子类重写 | 基于继承，类层级膨胀 |
| **装饰器模式** | 需要动态叠加功能，不想创建大量子类组合 | 包装同类接口对象，增强行为、不修改原有核心 | 多层包装调试难度上升 |
| **单例模式** | 资源实例不允许多次创建，全局访问冲突 | 构造私有，统一管控实例生命周期 | 测试困难；可能隐藏依赖 |

## 📊 三大流派模式分类

GoF 把 23 种模式分为三类，背后是三种应对变化的场景：

### 1. 创建型模式（隔离「对象创建过程」的变化）

**变化来源**：对象构造逻辑复杂、创建规则经常改动

**目标**：把「如何造对象」和「怎么使用对象」拆开

| 模式 | 核心思想 |
|------|----------|
| 工厂方法 | 定义创建对象的接口，让子类决定实例化哪个类 |
| 抽象工厂 | 提供一个创建一系列相关或相互依赖对象的接口 |
| 建造者 | 将复杂对象的构建与表示分离，使同样的构建过程可以创建不同的表示 |
| 原型 | 用原型实例指定创建对象的种类，通过拷贝这个原型来创建新的对象 |
| 单例 | 保证一个类只有一个实例，并提供一个全局访问点 |

### 2. 结构型模式（隔离「对象组合关系」的变化）

**变化来源**：对象之间组装、功能拓展方式多变

**目标**：稳定接口，灵活改变对象结构与包装方式

| 模式 | 核心思想 |
|------|----------|
| 适配器 | 将一个类的接口转换成客户希望的另一个接口 |
| 装饰器 | 动态地给一个对象添加一些额外的职责 |
| 代理 | 为其他对象提供一种代理以控制对这个对象的访问 |
| 桥接 | 将抽象部分与它的实现部分分离，使它们都可以独立地变化 |
| 组合 | 将对象组合成树形结构以表示"部分-整体"的层次结构 |
| 外观 | 为子系统中的一组接口提供一个一致的界面 |
| 享元 | 运用共享技术有效地支持大量细粒度的对象 |

### 3. 行为型模式（隔离「算法、职责、通信逻辑」的变化）

**变化来源**：运行逻辑、交互规则、算法策略频繁变更

**目标**：稳定调用方，把可变行为剥离出去

| 模式 | 核心思想 |
|------|----------|
| 策略 | 定义一系列算法，把它们封装起来，并使它们可以相互替换 |
| 模板方法 | 定义一个操作中的算法骨架，而将一些步骤延迟到子类中 |
| 观察者 | 定义对象间的一对多依赖，当一个对象状态改变时，所有依赖者都会收到通知 |
| 命令 | 将一个请求封装为一个对象，从而使你可以用不同的请求对客户进行参数化 |
| 迭代器 | 提供一种方法顺序访问聚合对象中的各个元素，而又不暴露该对象的内部表示 |
| 中介者 | 定义一个对象来封装一系列对象的交互 |
| 备忘录 | 在不破坏封装性的前提下，捕获一个对象的内部状态 |
| 状态 | 允许一个对象在其内部状态改变时改变它的行为 |
| 访问者 | 表示对某对象结构中各元素执行的操作 |
| 职责链 | 使多个对象都有机会处理请求，从而避免请求的发送者和接收者之间的耦合关系 |
| 解释器 | 给定一个语言，定义它的文法的一种表示，并定义一个解释器 |

## 💡 关键认知补充

**完整理解**：

> 只有预判未来该位置存在变化时，才主动划分边界隔离；如果确定这块逻辑永久不会变化，不需要额外抽象和模式。

**设计模式最大陷阱**：为了可扩展而提前扩展，引入无用抽象（过度设计）。

**变化预判 = 使用模式的前提**。

这也是为什么很多大厂工程师提倡：**先写直白代码，出现第一次变化时再重构引入抽象**，不要一开始堆砌模式。

## 🗣️ 语言选择建议

### 首选：Java / Kotlin

**理由**：
- 强静态类型、显式接口、类与继承语义清晰
- 强制要求显式写出抽象层，稳定层（抽象）、可变层（实现）代码物理上分开

当你手写一遍 Java 版策略、装饰器、模板方法，能直观看见：
- 抽象类 / 接口 = **稳定契约边界**
- 各种 Concrete 实现 = **被隔离的可变部分**

### ❌ 不建议初学直接用 Python

**原因**：
- 鸭子类型不需要显式抽象接口，经常可以省略稳定契约，代码很短，但边界被隐藏
- 新手容易只看懂运行效果，感受不到「抽象作为稳定屏障」的价值

**学习阶段原则**：越啰嗦、结构越显性的语言，越适合看清边界。工程落地阶段再换成简洁语言减少样板代码。

### 进阶建议

等你能够独立回答每个模式的「隔离什么变化」之后，再用 Go / Python / TS 重写同一批模式，你会获得第二层感悟：

| 语言特性 | 对模式的影响 |
|----------|-------------|
| 函数一等公民 | 策略模式经常直接被 lambda 替代 |
| Go 组合 + 隐式接口 | 很多基于继承的模式不再适用 |
| TS 类型体操 | 可以在类型层面实现更灵活的模式 |

## 📚 实操学习路线

### 阶段 1：地基（1~2 周）

- 吃透 SOLID，反复阅读开闭原则、里氏替换、依赖倒置
- **依赖倒置原则**是实现「稳定依赖抽象」最核心手段
  - 高层稳定模块不要依赖低层可变模块
  - 二者共同依赖抽象（稳定）

### 阶段 2：逐个研习模式（核心训练）

对每一个模式都照做：

```
场景 → 不用模式实现 → 感受痛点 → 引入模式重构 → 标记稳定/可变区域 → 总结弊端
```

1. **场景**：思考业务中什么场景会出现对应的变化痛点
2. **不用模式实现**：感受痛点——修改一处多处改动
3. **引入模式重构**：应用模式解决问题
4. **标记代码**：【稳定代码区域】【可变代码区域】【边界抽象】
5. **总结弊端**：什么场景下不该使用该模式

### 阶段 3：横向对比（非常重要）

很多模式目标相似，但隔离变化的维度不同，容易混淆：

| 对比组 | 隔离维度差异 |
|--------|-------------|
| 装饰器 vs 代理 | 装饰器关注功能增强；代理关注访问控制 |
| 策略 vs 模板方法 | 策略用组合替换算法；模板方法用继承固定骨架 |
| 适配器 vs 桥接 | 适配器解决接口不兼容；桥接分离抽象与实现 |

### 阶段 4：去套路化，回归实战

放下 23 个模式名词，拿到需求自问：

1. 未来哪些地方大概率变化？
2. 我该设置怎样的抽象边界隔离？
3. 有没有现成模式匹配这个场景？
4. 是否值得付出模式带来的复杂度成本？

做到这一步，才算掌握思想，而不是背诵模式。

---

## 🔖 总结

设计模式本质是一套经过验证的「**变化隔离方案库**」。

通过抽象定义稳定契约，将易变逻辑限制在契约实现侧，让系统改动收敛、不会向外扩散，达成开闭原则。
